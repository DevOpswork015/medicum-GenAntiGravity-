Kubernetes Cost Governance: The Autoscaling Trap

How we avoided a $100,000 cloud bill by fixing our Kubernetes autoscaling configuration

Introduction

We recently experienced a major cloud cost blowout in our Kubernetes cluster, with our monthly bill increasing by over 500%. The cause was a misconfigured autoscaling setup that led to an explosion of node creation, resulting in a huge bill.

Why it happens

The issue arose from a combination of factors, including a poorly designed scaling policy and inadequate monitoring. Our cluster was set up to scale based on CPU utilization, but the threshold was set too low, causing the cluster to scale up excessively.

Troubleshooting

To troubleshoot the issue, we used the following commands:

  • kubectl get hpa -o yaml to view the current Horizontal Pod Autoscaling (HPA) configuration
  • kubectl get nodes -o wide to view the current node list and their respective CPU utilization
  • kubectl top node to view the current CPU utilization of each node
kubectl get hpa -o yaml
# example output:
apiVersion: autoscaling/v2beta2
kind: HorizontalPodAutoscaler
metadata:
  name: example-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: example-deployment
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50

kubectl get nodes -o wide
# example output:
NAME           STATUS   ROLES    AGE   VERSION   INTERNAL-IP   EXTERNAL-IP   OS-IMAGE                KERNEL-VERSION   CONTAINER-RUNTIME
example-node   Ready    worker   1d    v1.23.5   10.0.0.1      192.0.2.1      Ubuntu 22.04.1 LTS   5.15.0-46-generic   containerd://1.6.8

Fixing the issue

To fix the issue, we updated our HPA configuration to use a more conservative scaling policy, and we also implemented additional monitoring and alerting to prevent similar issues in the future.

We used the following commands to update the HPA configuration:

  • kubectl patch hpa example-hpa -p='{"spec":{"metrics":[{"type":"Resource","resource":{"name":"cpu","target":{"type":"Utilization","averageUtilization":70}}}]}}' to update the HPA configuration to use a more conservative scaling policy

Lessons Learned

  • Regularly review and update your HPA configuration to ensure it is aligned with your current workload and scaling requirements
  • Implement additional monitoring and alerting to prevent similar issues in the future
