PostgreSQL Index Corruption: A Deep Dive into Detection, Mitigation, and Prevention

A real-world production incident, its root cause, and lessons learned
20 min read

Production Incident

I first noticed something was wrong when our monitoring system alerted us to a significant increase in query latency. The Grafana dashboard looked normal until I drilled down into the PostgreSQL metrics, where I saw a spike in index scan times.

Further investigation revealed that one of our critical indexes was corrupted, causing the database to perform full table scans instead of using the index. This resulted in a significant performance degradation, impacting our customers' experience.

The timeline of the incident was as follows: 10:00 AM - Alert triggered, 10:15 AM - Initial investigation, 10:30 AM - Root cause identified, 11:00 AM - Mitigation started, 12:00 PM - Incident resolved.

Architecture Overview

Our PostgreSQL database is hosted on Amazon RDS, with a multi-AZ deployment for high availability. The database is used by a microservices-based application, with multiple services connecting to it.

The infrastructure consists of:

  • 1 x PostgreSQL RDS instance (db.r5.large)
  • 1 x Read Replica (db.r5.large)
  • 1 x Application Load Balancer (ALB)
  • 2 x EC2 instances (c5.xlarge) running the microservices

Traffic flow: The ALB distributes traffic to the EC2 instances, which then connect to the PostgreSQL database.

Dependencies: The database is dependent on the EC2 instances, and the EC2 instances are dependent on the database.

Root Cause Analysis

The root cause of the incident was a combination of factors, including:

  • Insufficient index maintenance: We had not been regularly running VACUUM and ANALYZE on the database, leading to index bloat and corruption.
  • High write workload: Our application was experiencing a high volume of writes, which put additional stress on the database and contributed to the index corruption.
  • Lack of monitoring: We did not have adequate monitoring in place to detect index corruption, which delayed our response to the incident.

Investigation

To investigate the incident, we used the following tools and commands:

psql -U postgres -d mydatabase -c "CHECK TABLE mytable;"
pg_checksums -d mydatabase -t mytable
aws rds describe-db-instances --db-instance-identifier mydbinstance

We also analyzed the PostgreSQL logs using:

grep "index corruption" /var/log/postgresql/postgres.log

Reproduction

To reproduce the incident safely, you can use the following steps:

  1. Create a test database with a similar schema to the affected database.
  2. Insert a large amount of data into the database to simulate the high write workload.
  3. Disable index maintenance by setting autovacuum_enabled to off.
  4. Run a query that uses the corrupted index.

Production Fix

To fix the incident, we took the following steps:

  1. Ran VACUUM and ANALYZE on the affected table to rebuild the index.
  2. Enabled index maintenance by setting autovacuum_enabled to on.
  3. Monitored the database for any further issues.

The rollout plan for the fix was as follows:

  1. Rolled out the fix to the Read Replica first to test it.
  2. Monitored the Read Replica for 30 minutes to ensure the fix did not introduce any new issues.
  3. Rolled out the fix to the primary database instance.

The rollback strategy was to revert the changes made to the database configuration and restart the database instance.

Validation

To verify the success of the fix, we monitored the database metrics and query performance. We also ran a series of tests to ensure the index was functioning correctly.

Lessons Learned

The incident highlighted the importance of regular index maintenance and monitoring. We learned that:

  • Regularly running VACUUM and ANALYZE is crucial to preventing index corruption.
  • Monitoring the database for issues like index corruption is essential for detecting problems early.
  • A high write workload can contribute to index corruption, and we need to ensure our database is properly configured to handle it.

Best Practices

Based on the incident, we recommend the following best practices:

  • Regularly run VACUUM and ANALYZE on your database.
  • Monitor your database for issues like index corruption.
  • Ensure your database is properly configured to handle a high write workload.

Conclusion

In conclusion, the incident highlighted the importance of regular index maintenance and monitoring. By following the best practices outlined above, you can help prevent index corruption and ensure your database performs optimally.
