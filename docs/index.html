Unpacking the Causes of a Major Production Incident in Our Kubernetes Cluster

A deep dive into the technical details of a recent outage and the lessons learned
20 min read

Production Incident

I first noticed something was wrong when our Grafana dashboard showed a spike in latency for our API responses. The rollout of our latest feature had completed successfully just hours before, but it seemed that something had gone terribly wrong. Our customers began reporting errors, and our support team started receiving a flood of complaints.

The timeline of events was as follows: the feature rollout began at 02:00 AM, and by 03:00 AM, the first signs of trouble emerged. It took us approximately 2 hours to fully mitigate the issue, but not before a significant portion of our user base was affected.

Architecture Overview

Our application is built on a Kubernetes cluster, utilizing a combination of Docker containers and Helm charts for deployment. The infrastructure is hosted on AWS, leveraging services such as ELB for load balancing and RDS for database management. The traffic flow is as follows: users make requests to our ELB, which then routes them to one of our EC2 instances running the Docker container. The container communicates with our RDS database for data storage and retrieval.

Dependencies include the Kubernetes cluster itself, the Docker containers, the Helm charts for deployment, and the AWS services for hosting and load balancing.

Root Cause Analysis

After conducting a thorough investigation, we discovered that the root cause of the incident was a hidden assumption in our deployment script. The script assumed that all nodes in the cluster would have sufficient resources to handle the increased load following the feature rollout. However, this assumption proved incorrect, as one of our nodes was running low on resources, causing the containers to fail and resulting in the high latency observed.

The failure chain was as follows: the low resources on one node led to container failures, which in turn caused the ELB to route requests to other, already overloaded nodes, exacerbating the issue.

Investigation

We used a variety of tools to investigate the issue, including kubectl for inspecting the Kubernetes cluster, docker for examining container logs, and terraform for reviewing our infrastructure configuration. We also utilized promtool to query Prometheus metrics and curl to test API responses.

Example commands used during the investigation include:

  • kubectl get pods -o wide to view pod status and resource allocation
  • docker logs -f <container_id> to follow container logs in real-time
  • terraform state show <resource_name> to inspect the state of our infrastructure resources
  • promtool query 'rate(container_cpu_usage_seconds_total[1m])' to query the rate of CPU usage for our containers over the last minute

Reproduction

To safely reproduce the issue, one can follow these steps:

  1. Reduce the resources available to one node in the Kubernetes cluster
  2. Deploy a new feature or update to the application, simulating the conditions of the incident
  3. Monitor the cluster and application performance using tools like Prometheus and Grafana

Production Fix

The production fix involved updating our deployment script to account for variable resource availability across nodes. We also implemented a rollback strategy, allowing us to quickly revert to a previous version of the application if issues arose during deployment.

The risk associated with the fix was mitigated by thoroughly testing the updated script in a staging environment before applying it to production.

Validation

Success of the fix was verified through monitoring of the application and cluster performance post-deployment. Metrics such as latency and error rates were closely watched, and user feedback was collected to ensure that the issue had been fully resolved.

Lessons Learned

Several key lessons were learned from this incident. Firstly, the importance of thoroughly testing assumptions in deployment scripts cannot be overstated. Secondly, the need for robust monitoring and quick action in response to performance issues is critical. Lastly, continuous review and improvement of our infrastructure and application architecture are essential for preventing similar incidents in the future.

Best Practices

Best practices that can be applied to prevent similar incidents include:

  • Regularly reviewing and updating deployment scripts to reflect changing infrastructure and application requirements
  • Implementing comprehensive monitoring and logging to quickly detect and respond to performance issues
  • Conducting thorough testing of all changes in a staging environment before applying them to production

Conclusion

In conclusion, the production incident highlighted the importance of careful planning, thorough testing, and robust monitoring in preventing and mitigating outages. By applying the lessons learned from this incident and following best practices, we can work towards creating a more resilient and reliable application for our users.

As we move forward, a key question to consider is: How can we balance the need for rapid feature deployment with the necessity of ensuring the stability and performance of our application, especially in complex, distributed systems?
